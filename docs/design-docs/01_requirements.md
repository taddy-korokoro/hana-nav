## 1. プロダクト概要

「いつ・どこで・何が咲いているか」が一目で分かる、花畑スポット検索サービス。スマートフォンで撮った 1 枚の花写真から「これは何の花か → 近隣で見られる場所」まで導線が繋がる点が、既存の旅行ポータルや SNS との差別化ポイント。

### 1.1 コア価値提案

1. **見頃カレンダー × 地図検索**：「今・近場でどこに行けば花が見られるか」が 3 タップ以内で分かる。
2. **AI 植物判定 → 旅のしおり画像生成 → SNS 共有**：花の名前を知る・思い出をシェアする体験を 1 本に繋ぎ、自然な口コミ拡散を狙う。
3. **公式情報ベースの安心感**：全スポットに出典（公式 URL もしくは出典名）を必須化し、人手レビューを通った情報のみ公開する。

### 1.2 ビジネス上の前提

- 収益は MVP では追わない。リリース後に楽天アフィリエイト（スポット詳細＝楽天トラベル／花詳細＝楽天市場の 2 箇所）を限定的に設置。
- 月額ランニングコスト上限は ¥21,800（MAU 5,000 時の天井試算）。Gemini API のコスト爆発が最大リスクのため、レート制限と予算アラートで二重に防ぐ。

## 2. ペルソナ

ターゲットを「写真を SNS にあげる動機を持つファミリー／カップル」「花そのものへの関心が高い個人」に絞る。観光協会・自治体担当者は v2 以降の検討対象とし、MVP では含めない。

### P-1：旅行計画中のファミリー層（主要ペルソナ）

| 属性         | 内容                                                                                        |
| ------------ | ------------------------------------------------------------------------------------------- |
| 例           | 30 代後半・子持ち（小学生 1 人）・首都圏在住・年に 2〜3 回国内旅行                          |
| 旅行決定の場 | スマホ。Google 検索／Instagram から目的地を絞り、週末までに行先を決めたい                   |
| 困りごと     | 「今週末どこの花畑が見頃か」が情報源ごとにバラバラで、開花情報の鮮度を判断できない          |
| 来訪動機     | 「インスタ映え」より「子どもが喜ぶ＋無理のない移動」。混雑・私有地侵入は強い忌避要素        |
| 主に使う機能 | F-01 トップ（見頃マップ）／ F-02 検索／ F-03 詳細／ F-04 見頃カレンダー／ F-09 ブックマーク |

### P-2：写真と SNS が動機の旅行者（差別化ペルソナ）

| 属性         | 内容                                                                                                       |
| ------------ | ---------------------------------------------------------------------------------------------------------- |
| 例           | 20 代後半・1 人旅もしくはカップル旅行・Instagram / X で日常的に投稿                                        |
| 旅行決定の場 | 旅行先で偶然見つけた花を「これ何だっけ」と検索する。投稿用にひと工夫した画像が欲しい                       |
| 困りごと     | 花の名前が分からないと投稿の質が落ちる。手書きの「旅のしおり」風画像を作るのは面倒                         |
| 来訪動機     | 「映える写真 × その場で完結する撮影 → 投稿」体験                                                           |
| 主に使う機能 | F-05 AI 花判定／ F-06 旅のしおり生成／ F-03 詳細（関連スポット）／ F-08 ログイン（履歴を残したいユーザー） |

### P-3：花そのものに詳しくなりたい個人（コアファン）

| 属性         | 内容                                                                                              |
| ------------ | ------------------------------------------------------------------------------------------------- |
| 例           | 40〜60 代・ガーデニング／植物観察が趣味・地域の花スポットを定期的に巡る                           |
| 旅行決定の場 | 花の名前から逆引きで見られる場所を探す。シーズンを跨いだ計画を立てたい                            |
| 困りごと     | 花の和名・別名・品種名がバラバラで、検索ヒットがブレる                                            |
| 来訪動機     | 「あの花を見に行く」起点で行動。Wikipedia 的な濃い情報を求める                                    |
| 主に使う機能 | F-02 検索（花種類）／ 花の詳細（`/flowers/[id]`）／ F-10 簡易レビュー（WANT・知見の蓄積場として） |

## 3. 課題定義

### 3.1 ユーザー側の課題

1. **見頃情報の鮮度がバラバラ**：観光協会の公式 PDF、ブログ、SNS の投稿日が混在し、「今行けるか」の判断ができない。
2. **花の名前が分からない**：撮った花から検索する手段が「画像検索 → 不正確 → SNS で聞く」しかない。
3. **私有地・混雑情報が事前に分からない**：SNS 起点で「秘境」を訪れて住民とトラブルになる事例が増えている。
4. **思い出のアウトプット手段が手薄**：花スポットの写真は撮るが、SNS にあげる際の「一手間」が大きい。

### 3.2 サービス側で解決するアプローチ

| 課題           | 解決アプローチ（MVP）                                                                      |
| -------------- | ------------------------------------------------------------------------------------------ |
| 見頃情報の鮮度 | 月単位の「見頃」マスター（`best_season_start` / `best_season_end`）＋ 管理者人手レビュー   |
| 花の名前判定   | Gemini API による画像判定。総称＋品種名の 2 段返答で関連スポットへ自動リンク               |
| 私有地・混雑   | ピンは公式駐車場／入口に固定。コピーから「秘境」「穴場」を排除。マナー啓発文を全詳細に表示 |
| アウトプット   | Canvas API で縦長 1 枚の「旅のしおり」を生成し、Web Share API で SNS シェア                |

### 3.3 やらないこと（明示）

- リアルタイム開花情報の CGM 投稿（v2）
- 完全自動の見頃推定（気象データ連携）
- 観光協会向けダッシュボード／法人 API

## 4. 機能要件（F-01〜F-12）

優先度は `MUST`（MVP 必須・落とせない）／`WANT`（時間が許せば実装）の 2 段階。スケジュール遅延時の削減順序は `docs/specs/roadmap.md` を参照。

各機能には **目的**・**主要ユーザー導線**・**主要画面 / API**・**完了基準** を明記する。

### F-01 トップページ（見頃マップ × 検索 UI）／MUST

- **目的**：初訪問ユーザーが「今・近場で見頃の花畑」を 3 秒で把握できる入口を提供する。
- **主要導線**：トップ → 見頃カードをタップ → スポット詳細 ／ トップ → 花種類タイル → 花一覧。
- **主要画面**：`/`（Server Component で `flowers` / `spots` を集約取得）。
- **完了基準**：当月見頃のスポットが最大件数表示され、地図／リストの双方から詳細に遷移できる。

### F-02 スポット検索（エリア／花種類／見頃月）／MUST

- **目的**：「行先」起点・「花」起点のどちらでも辿り着けるよう、3 種の絞り込みを 1 画面で提供する。
- **主要導線**：`/spots` で `searchParams` を変更 → URL 共有可能な結果一覧 → カード → 詳細。
- **主要画面 / API**：`/spots`（`searchParams` に絞り込みを保持）／`/api/spots` ／ エリア別ショートカット `/areas/[prefecture_id]`。
- **完了基準**：47 都道府県すべてで絞り込めること。`searchParams` を直接 URL 入力しても再現できること。

### F-03 スポット詳細／MUST

- **目的**：「行くか／行かないか」をこのページだけで判断できるよう、地図・画像・関連花・マナー文をまとめる。
- **主要導線**：詳細到達 → 画像／関連花閲覧 → ブックマーク（F-09） or 公式サイト遷移。
- **主要画面 / API**：`/spots/[id]` ／`/api/spots/[id]`。`spots`＋`images`＋`spot_flowers`＋`flowers`＋`reviews` を読む。
- **完了基準**：未公開スポット（`is_published=false`）が一切表示されないこと。マナー啓発文と出典が必ず描画されること。

### F-04 見頃カレンダー（今月／来月の絞り込み）／MUST

- **目的**：時間軸を「今月／来月」に限定して直感的な絞り込みを提供する。
- **主要導線**：トップの「今月／来月」タブ ／ 検索画面の月セレクタ → 結果一覧。
- **主要画面 / 実装**：`lib/utils/seasonUtils.ts::isInBestSeason()` を全箇所で再利用（月またぎを正しく扱う）。
- **完了基準**：12〜1 月にまたがる花（例：椿）も正しく絞り込まれること。JST 基準で「今日」が判定されること（`lib/utils/dateUtils.ts`）。

### F-05 AI 植物判定／MUST（差別化）

- **目的**：撮った花の名前を即座に提示し、関連スポットへの送客動線を作る。
- **主要導線**：`/identify` で撮影／選択 → 結果画面（PictureThis 風 UI） → 関連スポット最大 5 件 → 詳細。
- **主要 API**：`POST /api/ai/identify-flower`（multipart）。Gemini `gemini-2.5-flash` を使用。
- **完了基準**：判定結果が「総称＋品種名＋花言葉＋豆知識」の構造で返ること。`flowers` マスターと 3 段階フォールバックでマッチすること。誤判定時は「自信なし」を明示できること。

### F-06 旅のしおり画像生成（SNS シェア）／MUST（差別化）

- **目的**：思い出を 1 枚の縦長画像にまとめ、自然な口コミ拡散を促す。
- **主要導線**：判定結果 → 「旅のしおりを作る」 → クライアント Canvas で画像生成 → Web Share API で SNS／保存。
- **主要画面**：`/identify/story`。クライアント完結（サーバー処理なし）。
- **完了基準**：iOS Safari／Chrome（モバイル）で画像が破綻なく生成されること。Web Share API 非対応時はダウンロードにフォールバックすること。

### F-07 レート制限（匿名 1 回／日、ログイン 3 回／日）／MUST

- **目的**：Gemini API のコスト爆発を防ぎつつ、ログインを促進する。
- **主要導線**：判定リクエスト時に `ai_usage_logs` を SELECT → カウント超過なら 429 を返し、リワード広告 or 翌日への案内を出す。
- **主要 API / テーブル**：`POST /api/ai/identify-flower` ／ `ai_usage_logs`（`user_id` か `anonymous_id` でカウント）。
- **完了基準**：匿名 ID は `localStorage` の UUID で安定すること。日跨ぎ（JST 0 時）でリセットされること。

### F-08 Supabase Auth（メール ＋ Google ログイン）／MUST

- **目的**：ブックマーク・レビュー・AI 判定回数上限緩和の前提を作る。
- **主要導線**：`/auth/signup` → メール確認 or Google OAuth → `/auth/callback` → 戻り先。
- **主要画面 / API**：`@supabase/ssr` を採用。`getUser()` でサーバー側判定。Middleware で保護ルートをリダイレクト。
- **完了基準**：未ログインで `/mypage/*` `/admin/*` にアクセスすると `/auth/login` に飛ぶこと。Cookie ハンドリングが `getAll()` / `setAll()` ペアであること。

### F-09 ブックマーク（行きたいリスト）／MUST

- **目的**：「あとで行く」を保存し、再訪・周遊計画の起点を作る。
- **主要導線**：詳細 → ハートボタン → `/mypage/bookmarks` で一覧。
- **主要 API / テーブル**：`POST /api/bookmarks` ／ `DELETE /api/bookmarks/[spot_id]` ／ `bookmarks` テーブル。
- **完了基準**：未ログイン時はログインへ誘導。退会後はカスケード論理削除でリストに残らないこと。

### F-10 簡易レビュー（★＋一言コメント）／WANT

- **目的**：訪問後の感想を残し、他ユーザーの判断材料にする。
- **主要導線**：詳細 → レビューを書く（ログイン必須） → 投稿。NG ワードフィルタを通過したものだけ表示。
- **主要 API / テーブル**：`POST /api/reviews`（Server Action 想定） ／ `reviews`。
- **完了基準**：退会済ユーザー（`profiles.deleted_at IS NOT NULL`）のレビューは「退会済ユーザー」表示で残ること。

### F-11 動画リワード広告で AI 追加利用／WANT

- **目的**：レート制限到達時に追加利用枠を提供しつつ、副次収益も得る。
- **主要導線**：429 応答 → リワード広告ダイアログ → 視聴完了 → `ai_usage_logs.reward_unlocked=true` → 上限 +5。
- **主要連携**：Google AdSense Reward。
- **完了基準**：広告枠の表示／非表示が環境変数で切り替えられること。ロジックは MVP に仕込み、UI 描画は時間次第で後置きしてよい。

### F-12 アフィリエイトリンク埋め込み／WANT

- **目的**：将来の収益化（楽天）と、検索動線（楽天市場で苗を探す等）の補強。
- **主要導線**：スポット詳細 → 楽天トラベルリンク／花詳細 → 楽天市場リンク。「広告」バッジ ＋ `rel="sponsored"` を必須化。
- **主要連携**：楽天アフィリエイト API（無料枠 1 req/sec）。
- **完了基準**：API キー未設定時は枠ごと非表示。景表法ステマ規制対応として全リンクに「広告」バッジが付くこと。

## 5. 非機能要件

### 5.1 性能

| 観点               | 目標                                                                                          |
| ------------------ | --------------------------------------------------------------------------------------------- |
| 初回ロード         | TOP / 検索 / 詳細の LCP 2.5s 以下（4G モバイル想定）                                          |
| 体感ナビゲーション | チケット 22 で `unstable_instant` を全画面適用し、再訪時の遷移は瞬間表示にする                |
| AI 判定レスポンス  | Gemini 呼び出し 5s 以内（タイムアウト 10s）。タイムアウト時は「もう一度お試しください」を表示 |
| しおり生成         | クライアント Canvas で 2s 以内。性能不足端末は 1080→720px に自動フォールバック                |

### 5.2 可用性

| 観点                  | 目標 / 方針                                                                   |
| --------------------- | ----------------------------------------------------------------------------- |
| 稼働率                | Netlify / Supabase の SLA に従う（MVP では独自 SLA を掲げない）               |
| 外部 API 障害時の挙動 | Gemini 障害時は判定機能のみ停止メッセージ。検索・詳細・ブックマークは継続稼働 |
| 障害復旧              | API キーの即時無効化手順（`docs/specs/operations.md`）を運用ドキュメント化済  |
| バックアップ          | Supabase 標準の日次バックアップを利用（MVP では追加対策なし）                 |

### 5.3 セキュリティ

| 観点       | ルール                                                                                                           |
| ---------- | ---------------------------------------------------------------------------------------------------------------- |
| 認証       | `@supabase/ssr`。サーバー判定は必ず `getUser()`、`getSession()` は信用しない                                     |
| 認可       | 全テーブルで RLS 有効。管理者判定は `profiles.role='admin'` ＋ `lib/utils/requireAdmin.ts`                       |
| 秘密情報   | `SUPABASE_SERVICE_ROLE_KEY` / `GEMINI_API_KEY` は `NEXT_PUBLIC_` を付けない。Client Component から import しない |
| 入力       | レビューは NG ワード辞書（`lib/ng-words.ts`）でフィルタ。すり抜けは管理画面から手動論理削除                      |
| 画像       | クライアントで `max-width:1024px` / JPEG 0.8 / 2MB 以下にリサイズしてから送信                                    |
| データ寿命 | 論理削除統一（`deleted_at`）。物理削除は禁止。全クエリで `WHERE deleted_at IS NULL` を必須                       |
| 個人情報   | プロフィール画像は自分の `auth.uid()` 配下フォルダのみ書き込み可（Storage RLS）                                  |

### 5.4 SEO

| 観点           | 目標                                                                     |
| -------------- | ------------------------------------------------------------------------ |
| メタデータ     | `metadata` / `generateMetadata` で動的生成。`<Head>` は使用禁止          |
| OGP            | デフォルト OG 画像 1200×630 を配置。スポット・花詳細は固有 OG 画像を出す |
| サイトマップ   | `app/sitemap.ts` で `spots` / `flowers` / 47 都道府県を全列挙            |
| robots         | `/admin/` `/api/` `/auth/` `/mypage/` を `disallow`                      |
| 構造化データ   | スポット詳細に `TouristAttraction` の JSON-LD（時間が許せば）            |
| 検索結果ページ | `searchParams` で組み合わせが膨大になるため `noindex` を推奨             |

### 5.5 コスト上限

| サービス    | MVP 時の月額上限             | 監視                                                          |
| ----------- | ---------------------------- | ------------------------------------------------------------- |
| Netlify     | Starter（無料）〜 Pro（$19） | Usage notifications ＋ Build minutes / Bandwidth 監視         |
| Supabase    | Free（500MB）〜 Pro（$25）   | Dashboard で DB サイズ監視                                    |
| Gemini API  | ¥3,000〜¥10,000              | Google Cloud で月予算 ¥5,000 超のアラート（必須）             |
| Google Maps | $200 無料枠内                | 超過時は Mapbox 無料枠への移行を検討                          |
| 合計        | **¥3,000〜¥21,800 を上限**   | 上記アラートいずれかが鳴った時点で SRE 的に手動でレート再設定 |

### 5.6 アクセシビリティ・対応端末

- モバイル First。タップターゲット 44px 以上、コントラスト比 4.5:1 以上を原則とする。
- 対応ブラウザ：iOS Safari／Android Chrome の最新 2 バージョン、PC Chrome／Safari／Edge の最新 2 バージョン。
- ダークモードは MVP では非対応（`@layer base` の `.dark` ブロックは未使用）。

## 6. システム化範囲・対象外

### 6.1 範囲（やる）

- **対応エリア**：日本国内（47 都道府県）。
- **対応言語 / TZ**：日本語のみ、JST 基準（DB は UTC 保管、表示時に変換）。
- **対応デバイス**：スマートフォン優先、PC でも閲覧可能。
- **データ投入**：管理者が Python スクレイパー 5 本（scrape→normalize→geocode→validate→upload）で初期 200 件投入し、人手レビューを経て公開する。
- **管理画面**：管理者ロール 1 種類（個別権限なし）。スポット・花・ユーザー・レビュー・AI ログ・画像の CRUD と公開可否管理を提供する。

### 6.2 対象外（やらない）

| 対象外                        | 理由 / 代替                                                |
| ----------------------------- | ---------------------------------------------------------- |
| ユーザー写真投稿（CGM）       | 著作権・モデレーション設計が重い。v2 で検討                |
| リアルタイム開花情報投稿      | 投稿/集計・スパム対策が重い。v2                            |
| プッシュ通知（PWA）           | iOS の PWA 通知制約。v2                                    |
| AI コンシェルジュ（旅程提案） | コスト見積もりが立たない。v3 以降                          |
| 法人向け API（観光協会）      | 営業・契約フローが必要。v3 以降                            |
| 多言語（英語・中文）          | MAU 5,000 で投資対効果が薄い。SEO で問題が顕在化したら検討 |
| カスタム SMTP（Resend）       | MVP では Supabase 同梱送信で運用。ユーザー定着後に切替     |
| 商標・法人登記                | MVP リリース後に確認                                       |

## 7. 用語集

| 用語                      | 説明                                                                                                                                                      |
| ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 見頃（best season）       | スポットごとの主要花の月単位開花期間。`spots.best_season_start` / `best_season_end`（1〜12）で表現。12→1 月などの月跨ぎは `isInBestSeason()` が判定       |
| スポット（spot）          | 花畑・公園・名所などの訪問先 1 件。`is_published=true` のもののみユーザーから見える                                                                       |
| 出典（source）            | スポット情報の根拠。`official_url` が NULL の場合は `source` 列（公式 URL もしくは情報元名）を必須化                                                      |
| 花マスター（flower）      | 花の種類カタログ。MVP では 30 種類以上を投入                                                                                                              |
| 別名（flower_alias）      | 花マスターのエイリアス。AI 判定結果（総称・品種名）とのマッチングに使用                                                                                   |
| しおり（story card）      | Canvas API で生成する縦長 1 枚の画像。SNS シェア用                                                                                                        |
| AI 判定                   | Gemini `gemini-2.5-flash` への画像入力で「総称＋品種名＋花言葉＋豆知識」を取得する処理                                                                    |
| 3 段階フォールバック      | `flowers.name` 完全一致 → `flower_aliases.alias` 完全一致 → AI が返した品種名で再エイリアス検索、の順で花を特定するロジック                               |
| レート制限                | `ai_usage_logs` で当日の判定回数をカウントし上限超過を弾く仕組み（匿名 1／ログイン 3／リワード視聴で +5）                                                 |
| 論理削除（soft delete）   | `deleted_at TIMESTAMPTZ` で行を「削除済み」にする方式。物理削除は禁止                                                                                     |
| カスケード論理削除        | 親（spots / flowers / profiles 等）が論理削除されたら、関連子レコードも DB トリガーで `deleted_at` を立てる仕組み                                         |
| 多態関連（images）        | `images` テーブルが `owner_type ('spot' \| 'flower')` ＋ `owner_id` で複数親を持つ設計。外部キー制約が張れないため、アプリ層 ＋ DB トリガーで整合性を保証 |
| Instant Navigation        | Next.js 16 の `unstable_instant` 機能。クライアントサイドの体感遷移を即座にする。チケット 22 で全画面適用                                                 |
| `@supabase/ssr`           | Next.js App Router 向け公式 Supabase クライアント。`@supabase/auth-helpers-nextjs` は deprecated                                                          |
| RLS（Row Level Security） | PostgreSQL の行単位アクセス制御。全テーブルで有効。SELECT ポリシーに `deleted_at IS NULL` を含めない（PostgREST RETURNING の仕様回避）                    |
| Service Role キー         | RLS をバイパスする全権キー。サーバーサイド／バッチに閉じて使用。Client Component から import しない                                                       |
