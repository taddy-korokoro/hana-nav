# ADR-002. AI 花判定の 3 段階フォールバックマッチング

| 項目       | 値                                                                 |
| ---------- | ------------------------------------------------------------------ |
| ステータス | Accepted（実装済み: `app/api/ai/identify-flower/route.ts`）        |
| 決定者     | プロジェクトリード                                                 |
| 関連 spec  | `docs/specs/ai-identify.md` / `docs/specs/database.md`             |
| 関連タスク | T06 外部 I/F 設計 / T08 API 詳細設計 / T09 AI 花判定処理フロー詳細 |

---

## 1. 背景

hana-nav は Gemini API が返す花名を `flowers` テーブルに紐付け、その花が見られる関連スポットを表示する。この「AI 出力 → DB マスター照合」がユーザー価値の中核で、ここでマッチしなければ「関連スポットが 0 件」という UX 悪化に直結する。

しかし、AI と DB の語彙は完全一致しない。

### 1-1. Gemini 側の返却特性

- `flower_name`（総称、マッチング用）と `flower_variety`（品種名、表示用）の両方を返すようプロンプトで指定している
- しかし、実際は以下のブレが起こる
  - 表記揺れ: 「桜」「サクラ」「cherry blossom」
  - 総称ではなく品種名だけを返してくる: 「ソメイヨシノ」（`flower_name` に品種名が入る）
  - 学名混じり: 「Prunus × yedoensis」

### 1-2. DB 側のマスター構造

- `flowers` テーブル: 総称単位で 1 レコード（例: `name = '桜'`）
- `flower_aliases` テーブル: 別名・表記揺れ・品種名を紐付ける多対 1 中間テーブル

つまり `flowers` だけで完全一致を狙うと、上記のブレをほぼ全て取りこぼす。かといって曖昧一致（LIKE / トライグラム）に頼ると誤マッチが増え、「桜」の画像から「桜草」が返る事故が起こりうる。

---

## 2. 決定

**Gemini 返却値 → `flowers` → `flower_aliases` の順で 3 段階の完全一致マッチングを行い、途中でヒットしたら以降はスキップする。全段階失敗時は `flower_master = null` で返す。**

| 段階 | 照合対象               | 検索キー                   | 意図                                                |
| ---- | ---------------------- | -------------------------- | --------------------------------------------------- |
| 1    | `flowers.name`         | `ai_result.flower_name`    | 総称ヒットの理想パス                                |
| 2    | `flower_aliases.alias` | `ai_result.flower_name`    | 表記揺れ・別名の吸収                                |
| 3    | `flower_aliases.alias` | `ai_result.flower_variety` | Gemini が総称を返さず品種名だけ返してきた場合の救済 |

### 2-1. なぜ 3 段階か

- **1 段階目（総称完全一致）は「理想パス」**。マスターと AI の語彙が揃っていればここで終わる。パフォーマンスも最良（インデックス直撃 1 クエリ）
- **2 段階目（alias 総称キー）は「表記揺れ吸収」**。マスター名が「桜」でも AI が「サクラ」「cherry blossom」を返した時に `flower_aliases` に事前登録した別名で拾う。alias 辞書のメンテナンスは admin 画面から行える
- **3 段階目（alias 品種名キー）は「最後の砦」**。Gemini が総称を返さず品種名だけ（例: 「ソメイヨシノ」）返してきたケースを拾う。品種名は `flower_aliases` に「桜 → ソメイヨシノ / ヤマザクラ / ...」の形で登録するため、alias テーブルの再利用で実装できる

段階を **これ以上増やすと誤マッチが増える** ため 3 段階で止める。詳細は §4-2 参照。

### 2-2. 打ち切り条件

各段階で `.maybeSingle()` を使い、1 件でもヒットした時点で以降の段階をスキップする。0 件確定した時のみ次段階に進む。

3 段階全て空だった場合:

- `flower_master = null`
- `flower_images = []`
- `recommended_spots = []`
- `ai_result` は返す（AI 判定精度モニタリング用）

### 2-3. カラム別 `deleted_at` フィルタ

各段階のクエリに `.is('deleted_at', null)` を必ず付与する。マスターが論理削除された場合（ADR-001）にゾンビマッチが起きないようにする。

## 3. 結果

### 3-1. 得られたもの

- **マッチ率の向上**: 1 段階のみの場合の推定マッチ率 40〜60%（Gemini の総称返却率に依存）に対し、3 段階まで含めると 80〜90% を狙える（alias 辞書の整備次第）
- **拡張性**: 新種花・新品種の追加は `flowers` + `flower_aliases` への INSERT だけで済む。マッチングロジックは変更不要
- **説明可能性**: どの段階でヒットしたかは実装上判別できるため、A/B テストや誤マッチ分析が容易

### 3-2. トレードオフ

| 項目                   | 内容                                                                                    |
| ---------------------- | --------------------------------------------------------------------------------------- |
| **alias 辞書のメンテ** | admin が別名を能動的に追加しないと 2・3 段階のヒット率が上がらない。運用負担がある      |
| **誤マッチのリスク**   | 品種名 alias に紛らわしい語（例: 「梅」に「ウメモドキ」）を入れると別属の花がマッチする |
| **DB クエリ回数増**    | 最悪 3 クエリ発行。ただし `flower_aliases_alias_idx` で 1 クエリ ~1ms、合計コスト微小   |

### 3-3. コストとのバランス

Gemini API の 1 呼び出しコストは（`gemini-2.5-flash` / 画像 1024px 圧縮済み）約 $0.001。マッチング側の DB クエリコストは Supabase Free 枠内で無料。

- 3 段階マッチングで **追加 DB クエリ最大 2 本** 発生するが、Gemini 呼び出しコストと比較すると 3 桁小さい
- マッチ率を上げれば「AI 判定失敗 → 再撮影 → 再度 Gemini 呼び出し」というループを減らせるため、**総合的には Gemini コストを下げる** 方向に働く
- キャッシュ戦略（`docs/design-docs/09_ai-flow.md` §3）で同一画像の重複呼び出しはゼロ化されるため、マッチ率向上の副作用として発生する「再撮影削減」の効果はキャッシュ効果と比べて相対的に小さいが、**UX 上の意味は大きい**（1 度でヒットしないと離脱率が上がる）

---

## 4. 検討した代替案

### 案 A: 1 段階（`flowers.name` 完全一致のみ）

**却下理由**: Gemini の表記揺れ・品種名混入で推定マッチ率が 40〜60% に留まり、「関連スポット 0 件」体験が過半数のリクエストで発生する。MVP の中核体験としては不可。

### 案 B: 部分一致・LIKE 検索

**却下理由**: 「桜」で LIKE すると「桜草（サクラソウ）」「山桜桃（ユスラウメ）」がヒットしうる。品種の異なる別属を誤マッチさせるとユーザー価値がむしろ下がるため見送り。トライグラム類似検索も同様のリスク。

### 案 C: エンベディング + ベクトル検索（pgvector）

**却下理由**:

- Supabase pgvector 拡張の導入・埋め込み計算コストが 4 週間ローンチには重い
- 「桜」と「桜草」のようにテキスト距離が近い異属花を分離する精度を担保しにくい
- 現状の 3 段階マッチングでマッチ率 80〜90% を狙える見込みがあり、費用対効果でオーバースペック

v2 拡張で MVP マッチ率が伸び悩んだ場合の選択肢として残す（`docs/specs/roadmap.md` 参照）。

### 案 D: 4 段階以上（例: 品種名 → `flowers.name` 部分一致を追加）

**却下理由**: 段階を増やすほど誤マッチ確率が段階的に上がる。3 段階までは「完全一致 + 事前定義した alias」で誤マッチ耐性を保てるが、4 段階目以降は「AI 出力を推測で解釈する」領域に入り、責任分界が曖昧になる。
